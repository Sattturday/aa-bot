### User Story: Валидация API через Zod

**ID:** FEAT-005
**Название:** Добавить валидацию входных данных API с использованием Zod

---

### Описание

Как разработчик,
я хочу валидировать входные данные API на уровне схем,
чтобы предотвратить попадание некорректных данных в бизнес-логику и базу данных.

---

### Контекст

В текущей реализации контроллеры (`messagesController`, `urlsController`, `settingsController`, `adminsController`) принимают данные без валидации. Это создаёт риск ошибок, неконсистентных данных и усложняет отладку.

Исключение — `groupsController`, где уже есть частичная ручная проверка.

---

### Acceptance Criteria

1. **Добавлены Zod-схемы для всех ресурсов:**

   * `groupSchema.ts` — логика перенесена из `groupsController`
   * `messageSchema.ts` — `key: string`, `value: non-empty string`
   * `urlSchema.ts` — `key: string`, `value: valid URL`
   * `settingSchema.ts` — `key: string`, `value: string`
   * `adminSchema.ts` — `telegram_id: string`, только цифры

2. **Создан middleware `validateBody(schema)`**

   * валидирует `req.body` через `schema.safeParse`
   * при ошибке возвращает `400` с деталями
   * при успехе заменяет `req.body` на валидированные данные

3. **Все контроллеры используют middleware**

   * валидация вынесена из контроллеров в роутеры
   * контроллеры больше не содержат проверок входных данных

4. **Поведение API:**

   * невалидные данные → `400 Bad Request`
   * валидные данные → передаются в контроллер без изменений

---

### Definition of Done

* Установлена зависимость `zod`
* Создана папка `src/api/validation/` с 5 схемами + middleware
* Все соответствующие роуты используют `validateBody`
* Удалена ручная валидация из `groupsController`
* Проект компилируется без ошибок
* Проверка:

  * невалидный payload → 400
  * валидный payload → успешный ответ

---

### Технические заметки

* Использовать `safeParse`, а не `parse` (без исключений)
* Ошибки возвращать через `error.flatten()`
* Не дублировать типы — при необходимости выводить типы из Zod (`z.infer`)

---

### Scope

**Входит:**

* схемы
* middleware
* интеграция в роутеры

**Не входит:**

* валидация query/params
* изменение бизнес-логики контроллеров

---

### Риски

* Возможные breaking changes, если фронт отправляет невалидные данные
* Неполный перенос логики из `groupsController`

---

### Приоритет

Средний — влияет на стабильность API, но не блокирует текущую работу
