# Конституция AA Bot

## Основные принципы

### I. Русскоязычный пользовательский опыт

Весь текст бота, подписи кнопок, сообщения об ошибках и
инлайн-ответы ОБЯЗАНЫ быть на русском языке. Интерфейс
админ-панели МОЖЕТ использовать английский для служебных
подписей, но ОБЯЗАН отображать редактируемый пользователем
контент (сообщения, названия групп, описания) на русском
языке в том виде, в котором он сохранён.

Обоснование: бот обслуживает русскоязычное сообщество АА;
смешение языков подрывает доверие и удобство использования.

### II. Контент из базы данных

Весь пользовательский контент (сообщения, URL-адреса, данные
групп, настройки) ОБЯЗАН читаться из SQLite через слой
репозиториев. Хардкод в обработчиках ЗАПРЕЩЁН. Статические
файлы в `src/data/` существуют исключительно для начального
наполнения (seed) и НЕ ЯВЛЯЮТСЯ источником истины в рантайме.

Новые группы, сообщения и URL-адреса ОБЯЗАНЫ управляться
через админ API без изменения кода или перезапуска бота.

### III. Когерентность кеша

Каждая запись через админ API, изменяющая данные, потребляемые
ботом, ОБЯЗАНА вызвать соответствующую функцию `invalidate*()`
в `src/db/dataProvider.ts` до возврата успешного ответа.

Без инвалидации бот отдаёт устаревшие данные до 60 секунд.
При добавлении новых кешируемых источников НЕОБХОДИМО
использовать класс `DataCache` и предоставить функцию
инвалидации.

### IV. Целостность навигационного стека

Навигационный стек пользователя (`pushToStack` / `popFromStack`)
является единственным механизмом поведения кнопки «Назад».

Каждый обработчик действия, переводящий пользователя на новый
экран, ОБЯЗАН положить ключ текущего экрана в стек перед
отрисовкой нового. Обработчики НЕ ДОЛЖНЫ манипулировать
стеком других пользователей или обходить стек для навигационных
ярлыков.

### V. Минимальная сложность

Прямые, простые реализации предпочтительнее абстракций.
Запрещены: feature-флаги, ORM, dependency injection. SQLite
с синхронным API `better-sqlite3` — единственное хранилище
данных.

Новые зависимости ОБЯЗАНЫ быть обоснованы. Бот — это
однопроцессное Node.js-приложение и ОБЯЗАН оставаться
развёртываемым как один Docker-контейнер с примонтированной
директорией `data/`.

Задачи НЕ ДОЛЖНЫ усложняться сверх того, что было запрошено.
Если задача простая — реализация ОБЯЗАНА быть простой.
Не добавлять лишних абстракций, обёрток или «на будущее».

### VI. Документация на русском в стиле Google Docstring

Вся проектная документация (спецификации, планы, описания
задач, комментарии в коде) ОБЯЗАНА быть написана на русском
языке.

Формат документирования функций и модулей — Google Docstring:

```
/**
 * Краткое описание функции.
 *
 * Развёрнутое описание поведения, если необходимо.
 *
 * Args:
 *   paramName: Описание параметра.
 *   anotherParam: Описание другого параметра.
 *
 * Returns:
 *   Описание возвращаемого значения.
 *
 * Raises:
 *   Error: Условие, при котором выбрасывается ошибка.
 *
 * Example:
 *   const result = myFunction('input');
 */
```

Обоснование: единый стиль документации на русском языке
обеспечивает согласованность и снижает когнитивную нагрузку
при работе с кодовой базой.

### VII. Ветвление по фичам

Каждая новая фича ОБЯЗАНА разрабатываться в отдельной
git-ветке. Формат имени ветки: `feature/<краткое-описание>`.
Слияние в `main` — только через pull request.

Обоснование: изоляция изменений снижает риск конфликтов
и упрощает код-ревью.

### VIII. Асинхронный код

Новый код ОБЯЗАН использовать `async/await` для всех
асинхронных операций: сетевые вызовы, Telegram API,
файловый ввод-вывод. Колбэки и `.then()`-цепочки
ЗАПРЕЩЕНЫ для нового кода.

Обоснование: `async/await` улучшает читаемость и упрощает
обработку ошибок через `try/catch`.

### IX. Обязательное тестирование

Каждое изменение кода ОБЯЗАНО сопровождаться тестами.
Тесты ОБЯЗАНЫ запускаться после каждого изменения в коде
командой `npm test`.

При добавлении новой функциональности ОБЯЗАНЫ быть написаны:
- Юнит-тесты для бизнес-логики и утилит
- Интеграционные тесты для API-эндпоинтов

Обоснование: тесты — единственная гарантия того, что
изменения не ломают существующую функциональность.

### X. Линтинг

Линтер ОБЯЗАН запускаться перед каждым коммитом. Код
НЕ ДОЛЖЕН содержать ошибок и предупреждений линтера.

Для бота: `npm run lint` (когда настроен).
Для админ-панели: `cd admin && npm run lint`.

Обоснование: автоматическая проверка стиля кода
предотвращает накопление технического долга.

### XI. Именованные аргументы

Функции с более чем двумя параметрами ОБЯЗАНЫ принимать
один объект с именованными полями вместо позиционных
аргументов.

Пример — правильно:
```typescript
function createGroup(params: {
  name: string;
  type: 'aa' | 'alanon';
  address: string;
}): void { ... }
```

Пример — неправильно:
```typescript
function createGroup(
  name: string,
  type: string,
  address: string
): void { ... }
```

Обоснование: именованные аргументы устраняют ошибки
перепутывания порядка параметров и делают вызовы
самодокументируемыми.

## Технологические ограничения

- **Среда выполнения**: Node.js 18+, TypeScript (strict mode),
  компиляция в `dist/` через `tsc`
- **Фреймворк бота**: Telegraf 4.x (Telegram Bot API)
- **HTTP-сервер**: Express 5.x, запускается в том же процессе,
  что и бот
- **База данных**: SQLite через `better-sqlite3`, режим WAL,
  внешние ключи включены. Файл БД: `data/bot.db`
- **Админ-панель**: React + Vite + Tailwind CSS, собирается
  в `admin/dist/`, раздаётся Express как статика
- **Аутентификация**: валидация `initData` Telegram Mini App
  через HMAC-SHA256 → JWT (срок действия 1 час). Доступ
  только для администраторов из таблицы `admins`
- **Тестирование**: тесты запускаются через `npm test`
  после каждого изменения кода

## Рабочий процесс разработки

- `npm run dev` — разработка бота (TypeScript watch + nodemon)
- `cd admin && npm run dev` — разработка админ-панели (Vite)
- Админ-панель ОБЯЗАНА быть собрана (`cd admin && npm run build`)
  перед деплоем; Express раздаёт файлы из `admin/dist/`
- Новые фичи — в отдельных ветках `feature/<описание>`
- После изменений в коде ОБЯЗАНЫ быть запущены тесты и линтер
- Бот отправляет стартовое сообщение на `TG_ID` при запуске —
  используйте это для подтверждения успешного деплоя
- Docker-сборка ОБЯЗАНА давать один образ; директория `data/`
  ОБЯЗАНА быть примонтирована как volume для сохранения базы
  SQLite между перезапусками контейнера

## Управление

Данная конституция определяет неизменяемые архитектурные
решения проекта AA Bot. Для внесения поправок необходимо:

1. Задокументировать предлагаемое изменение и его обоснование
2. Обновить этот файл
3. Увеличить версию по правилам semver:
   - MAJOR — удаление или переопределение принципов
   - MINOR — добавление новых принципов или секций
   - PATCH — уточнения, исправления формулировок
4. Проверить согласованность с артефактами в `.specify/templates/`

Все реализации фич и код-ревью ОБЯЗАНЫ проверять соответствие
этим принципам. Для оперативного руководства по разработке
используйте `CLAUDE.md`.
