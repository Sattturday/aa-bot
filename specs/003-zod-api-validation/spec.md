# Feature Specification: Валидация API через Zod

**Feature Branch**: `003-zod-api-validation`
**Created**: 2026-03-27
**Status**: Draft
**Input**: User description: "Добавить валидацию входных данных API с использованием Zod"

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Невалидные данные отклоняются на уровне роутера (Priority: P1)

Как разработчик, я хочу, чтобы API автоматически отклонял некорректные входные данные до того, как они попадут в бизнес-логику, и возвращал понятное сообщение об ошибке.

**Why this priority**: Это ядро фичи. Без этого смысл валидации отсутствует. Позволяет немедленно получить value: ни один невалидный payload больше не проходит в контроллеры.

**Independent Test**: Отправить POST-запрос с невалидным телом на любой роут (например, пустое поле, неверный URL, нецифровой telegram_id) — получить 400 с описанием ошибки.

**Acceptance Scenarios**:

1. **Given** API-роут принимает `key` и `value`, **When** клиент отправляет тело с пустым `value`, **Then** система возвращает `400 Bad Request` с деталями о нарушении.
2. **Given** роут создания URL принимает валидный URL в `value`, **When** клиент отправляет произвольную строку вместо URL, **Then** система возвращает `400 Bad Request`.
3. **Given** роут администраторов принимает `telegram_id` как строку из цифр, **When** клиент отправляет строку с буквами, **Then** система возвращает `400 Bad Request`.
4. **Given** все поля валидны, **When** клиент отправляет корректный payload, **Then** система обрабатывает запрос и возвращает успешный ответ.

---

### User Story 2 — Контроллеры очищены от ручной валидации (Priority: P2)

Как разработчик, я хочу, чтобы контроллеры отвечали только за бизнес-логику, а не за проверку входных данных — это упрощает поддержку кода.

**Why this priority**: Это качественное улучшение: единственное место для правил валидации, меньше дублирования, проще читать и тестировать. Не блокирует безопасность, но снижает технический долг.

**Independent Test**: Открыть любой контроллер (groups, messages, urls, settings, admins) — в них не должно быть кода, проверяющего формат или наличие полей входящего запроса.

**Acceptance Scenarios**:

1. **Given** контроллер `groupsController` содержит ручные проверки входных данных, **When** валидация перенесена в middleware, **Then** в контроллере не остаётся кода, проверяющего поля запроса.
2. **Given** несколько контроллеров с разной логикой, **When** все они подключают общий middleware, **Then** правила валидации для каждого ресурса определены ровно в одном месте.

---

### Edge Cases

- Пустое тело `{}` — обязательные поля отсутствуют → 400 Bad Request с описанием отсутствующих полей.
- `null` или нестроковое значение для ожидаемой строки — не соответствует типу → 400 Bad Request.
- Дополнительные неизвестные поля в теле запроса — молча удаляются (strip), контроллер не получает лишних данных, запрос продолжается.
- Слишком длинное значение `value` — верхней границы нет; поле валидируется только на непустоту (`min 1`).

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система ДОЛЖНА отклонять запросы с невалидным телом до передачи в контроллер, возвращая `400 Bad Request`.
- **FR-002**: Ответ при ошибке валидации ДОЛЖЕН содержать структурированное описание нарушений (какое поле, какое правило нарушено).
- **FR-003**: Система ДОЛЖНА валидировать `key` как непустую строку в теле запроса для ресурса groups (createGroup). Для ресурсов messages, urls, settings — `key` передаётся как path-параметр и не входит в тело запроса; валидация body-полей для них не включает `key`.
- **FR-004**: Система ДОЛЖНА валидировать `value` как непустую строку для ресурсов messages, settings, groups.
- **FR-010**: Система ДОЛЖНА валидировать поле `phone` группы как строку формата `+7 (XXX) XXX-XX-XX` (regex `/^\+7 \(\d{3}\) \d{3}-\d{2}-\d{2}$/`), если поле присутствует.
- **FR-011**: Система ДОЛЖНА валидировать поле `days` элемента расписания как непустой массив, каждый элемент которого — одно из значений: `Пн`, `Вт`, `Ср`, `Чт`, `Пт`, `Сб`, `Вс`.
- **FR-012**: Система ДОЛЖНА валидировать поле `time` элемента расписания как строку формата `ЧЧ:ММ` (regex `/^\d{2}:\d{2}$/`).
- **FR-005**: Система ДОЛЖНА валидировать `value` как корректный URL для ресурса urls.
- **FR-006**: Система ДОЛЖНА валидировать `telegram_id` как строку, состоящую только из цифр, для ресурса admins.
- **FR-007**: При успешной валидации контроллер ДОЛЖЕН получать только валидированные данные, не требуя дополнительных проверок.
- **FR-008**: Контроллеры НЕ ДОЛЖНЫ содержать кода для проверки формата или наличия полей входящего запроса.
- **FR-009**: Зависимость для схемной валидации ДОЛЖНА быть добавлена в проект.

### Key Entities

- **Схема валидации**: Набор правил для конкретного ресурса (поля, типы, ограничения). Пять схем: для групп, сообщений, URL, настроек, администраторов.
- **Middleware валидации**: Компонент, применяющий схему к входящему запросу. Один общий компонент, параметризуемый схемой.
- **Ошибка валидации**: Структурированный ответ с описанием нарушений, возвращаемый клиенту при отклонении запроса.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 100% API-роутов, принимающих тело запроса, защищены валидацией — включая `PUT /groups/:id` (partial-схема). Невалидный payload не может попасть в контроллер.
- **SC-002**: Любой невалидный запрос получает ответ `400` с человекочитаемым описанием ошибки.
- **SC-003**: Ни один контроллер не содержит кода проверки формата входных данных — логика валидации консолидирована в схемах.
- **SC-004**: Проект компилируется без ошибок после внесения изменений.
- **SC-005**: Существующий функциональный сценарий (валидный payload → успешный ответ) продолжает работать без изменений.

## Clarifications

### Session 2026-03-27

- Q: Как обрабатываются дополнительные неизвестные поля в теле запроса? → A: Strip — неизвестные поля молча удаляются из `req.body`, запрос продолжается нормально.
- Q: Должен ли `PUT /groups/:id` (updateGroup) получить body-валидацию? → A: Да, через partial-схему (все поля опциональны).
- Q: Каковы форматы для дней расписания, времени и телефона? → A: Дни — только сокращения дней недели (`Пн`, `Вт`, `Ср`, `Чт`, `Пт`, `Сб`, `Вс`); время — формат `ЧЧ:ММ` (regex `^\d{2}:\d{2}$`); телефон — формат `+7 (XXX) XXX-XX-XX`.

## Assumptions

- Валидация распространяется только на тело запроса (`req.body`); query-параметры и path-параметры не входят в область применения.
- Бизнес-логика контроллеров не изменяется — только убирается ручная валидация входных данных.
- Фронтенд (admin panel) отправляет корректные данные в штатном режиме; 400-ответы при невалидных данных являются ожидаемым поведением для защиты от некорректных запросов.
- Типы TypeScript для тел запросов выводятся из схем валидации без дублирования определений.
- Ограничения на длину полей не определены явно — используются разумные значения по умолчанию (непустая строка без верхнего предела, если не указано иное).
