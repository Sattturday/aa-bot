# Исследование: Централизация текстов через i18n-слой

## Решение 1: Источник истины для текстов в рантайме

**Решение**: Использовать гибридный резолвер: сначала runtime-значение из текущего data provider (данные БД/кэш), при отсутствии значения брать из локального i18n-каталога.

**Обоснование**: Это сохраняет существующую возможность редактирования сообщений через админ API и одновременно убирает строки из хендлеров, что соответствует целям фичи.

**Альтернативы**:
- Только локальный словарь в коде. Отклонено: противоречит принципу «контент из БД» и ломает существующий админ-поток редактирования.
- Только БД без локального словаря. Отклонено: нет безопасного дефолта при неполном наборе данных в БД.

## Решение 2: Формат ключей сообщений

**Решение**: Ввести стабильные строковые ключи в точечной нотации (`section.item`) и выводить union-тип ключей из структуры локального каталога.

**Обоснование**: Точечная нотация делает ключи читаемыми и масштабируемыми, а вывод типа из каталога даёт compile-time контроль без ручного перечисления.

**Альтернативы**:
- Плоские ключи (`welcome_message`, `group_not_found`). Отклонено: хуже структурируемость при росте каталога.
- Ручной union-тип ключей. Отклонено: высокий риск рассинхронизации со словарём.

## Решение 3: Поведение fallback при пропуске ключа

**Решение**: Если ключ отсутствует в обоих источниках или значение пустое, логировать диагностическое событие и возвращать сам ключ как безопасный fallback.

**Обоснование**: Возврат ключа позволяет не ломать пользовательский сценарий и быстро увидеть проблемное место, а логирование даёт трассировку для исправления.

**Альтернативы**:
- Возвращать пустую строку. Отклонено: пользователь видит «пустые» сообщения, проблема обнаруживается поздно.
- Бросать исключение. Отклонено: может прерывать пользовательский флоу.

## Решение 4: Где размещать резолвер и словарь

**Решение**: Разместить модуль в `src/i18n/` с публичным API `t(key)` и отдельными файлами для словаря и типов.

**Обоснование**: Явное выделение слоя упрощает миграцию строк, дальнейшее расширение и не смешивает текстовую инфраструктуру с хендлерами.

**Альтернативы**:
- Встроить `t()` в `src/utils/utils.ts`. Отклонено: нарушает разделение ответственности и усложняет повторное использование.
- Держать словарь в `src/data/messages.ts` без отдельного API. Отклонено: не решает задачу унифицированного доступа и типобезопасности.

## Решение 5: Объём миграции в рамках текущей фичи

**Решение**: Перенести все пользовательские строки из `src/handlers/*` и `src/utils/utils.ts` на `t(key)`, включая fallback-сообщения и подписи, кроме явно динамических строк, формируемых из runtime-данных (например, расписание групп).

**Обоснование**: Это закрывает основной функциональный scope и не требует переписывания доменной логики формирования динамического контента.

**Альтернативы**:
- Частичная миграция только часто используемых экранов. Отклонено: оставляет дублирование и смешанный стиль.
- Полная нормализация всех динамических шаблонов в этой же задаче. Отклонено: избыточно для текущего приоритета и повышает риск регрессии.
