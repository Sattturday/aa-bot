# Feature Specification: Централизация текстов через i18n-слой

**Feature Branch**: `008-centralize-i18n-texts`  
**Created**: 2026-04-01  
**Status**: Draft  
**Input**: User description: "Централизация текстов через i18n-слой: хранить все тексты бота централизованно через t(key), типизировать ключи, вынести тексты в отдельный слой и заменить хардкод в handlers/utils с fallback и готовностью к БД"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Единая точка доступа к текстам (Priority: P1)

Как разработчик, я хочу получать пользовательские тексты только через единый вызов по ключу, чтобы убрать дублирование строк и снизить риск расхождений между экранами.

**Why this priority**: Это базовая ценность фичи. Без единого доступа цель централизации не достигается.

**Independent Test**: Проверить основные сценарии бота (`/start`, навигация по кнопкам) и подтвердить, что пользовательские тексты приходят из централизованного словаря, а не из хардкода в обработчиках.

**Acceptance Scenarios**:

1. **Given** в системе определён текстовый ключ, **When** обработчик запрашивает текст по этому ключу, **Then** возвращается единый текст из централизованного источника.
2. **Given** один и тот же текст используется в нескольких местах, **When** его значение обновлено в централизованном словаре, **Then** новое значение применяется во всех этих местах.

---

### User Story 2 - Защита от ошибок в ключах (Priority: P2)

Как разработчик, я хочу, чтобы система не позволяла использовать несуществующие ключи сообщений, чтобы находить ошибки на этапе сборки, а не в рантайме.

**Why this priority**: Это напрямую снижает количество дефектов и регрессий при изменении текстов.

**Independent Test**: Попробовать обратиться к несуществующему ключу сообщения и убедиться, что проверка проекта завершается ошибкой до запуска.

**Acceptance Scenarios**:

1. **Given** разработчик использует корректный ключ сообщения, **When** выполняется проверка проекта, **Then** ошибок по ключу нет.
2. **Given** разработчик использует отсутствующий ключ сообщения, **When** выполняется проверка проекта, **Then** проект сообщает о недопустимом ключе и не проходит проверку.

---

### User Story 3 - Безопасное поведение при неполных данных (Priority: P3)

Как разработчик и оператор системы, я хочу безопасный fallback при отсутствии текста, чтобы бот не ломал пользовательский сценарий и сигнализировал о проблеме в логах.

**Why this priority**: Сценарий менее частый, но критичен для устойчивости и быстрой диагностики.

**Independent Test**: Удалить или скрыть одно значение в словаре и проверить, что бот возвращает безопасный fallback и пишет диагностическое сообщение.

**Acceptance Scenarios**:

1. **Given** значение для ключа отсутствует, **When** система запрашивает текст, **Then** возвращается безопасное fallback-значение вместо падения.
2. **Given** значение для ключа отсутствует, **When** срабатывает fallback, **Then** система фиксирует диагностическую запись для последующего анализа.

---

### Edge Cases

- Ключ существует, но значение пустое: система должна применять то же fallback-поведение, что и при отсутствии значения.
- В коде остались единичные хардкоженные строки: такие места считаются дефектом миграции и должны быть выявлены при ревью.
- Ключ был переименован, но одно из обращений не обновлено: проверка проекта должна обнаружить несоответствие до запуска.
- Временная недоступность внешнего источника текстов в будущем расширении: система должна продолжать работать на локальном словаре.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система ДОЛЖНА предоставлять единый интерфейс получения пользовательского текста по ключу.
- **FR-002**: Система ДОЛЖНА хранить пользовательские тексты в централизованном словаре, доступном для всех обработчиков и утилит.
- **FR-003**: Все пользовательские тексты, отправляемые ботом, ДОЛЖНЫ извлекаться через единый интерфейс, а не из локально захардкоженных строк.
- **FR-004**: Система ДОЛЖНА ограничивать допустимые ключи сообщений заранее определённым набором ключей словаря.
- **FR-005**: Система ДОЛЖНА сигнализировать об использовании недопустимого ключа на этапе проверки проекта до запуска приложения.
- **FR-006**: При отсутствии текста по ключу система ДОЛЖНА возвращать безопасное fallback-значение.
- **FR-007**: При каждом fallback-событии система ДОЛЖНА записывать диагностическую информацию в лог.
- **FR-008**: Централизованный слой текстов ДОЛЖЕН поддерживать будущую замену или переопределение источника сообщений без изменения вызовов в бизнес-сценариях.
- **FR-009**: Изменение ДОЛЖНО сохранить текущее пользовательское поведение основных сценариев бота при наличии корректных данных.

### Key Entities *(include if feature involves data)*

- **Message Catalog**: Централизованный набор пользовательских текстов, организованный по стабильным ключам.
- **Message Key**: Уникальный идентификатор текста, используемый для обращения к значению в каталоге.
- **Text Resolver**: Единая точка получения текста по ключу с проверкой ключа и fallback-поведением.
- **Fallback Event**: Событие, фиксирующее случай отсутствия значения и диагностические данные для анализа.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: В 100% проверенных основных пользовательских сценариев (`/start`, навигация по кнопкам) тексты выдаются через централизованный слой без регрессии поведения.
- **SC-002**: Для 100% обращений к несуществующим ключам проект не проходит проверку до запуска.
- **SC-003**: При искусственно удалённом значении система в 100% попыток возвращает безопасный fallback и пишет диагностическую запись.
- **SC-004**: Изменение текста в централизованном каталоге отражается во всех связанных местах без правок в обработчиках.

## Assumptions

- На первом этапе используется один язык интерфейса, расширение до мультиязычности выполняется отдельно.
- Тексты, которые не отображаются пользователю (например, чисто технические внутренние сообщения), не входят в обязательный объём миграции.
- Основным критерием полноты миграции считается отсутствие пользовательских хардкоженных строк в обработчиках и утилитах.
- Будущий внешний источник текстов должен поддерживать те же ключи и fallback-политику, что и локальный словарь.
