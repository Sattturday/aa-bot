# Feature Specification: Рефакторинг обработчиков Telegram

**Created**: 2026-03-27

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Централизованная регистрация обработчиков (Priority: P1)

Как разработчик, я хочу регистрировать новую категорию обработчиков через единый вызов функции, а не дублировать один и тот же блок кода для каждой категории вручную.

**Why this priority**: Это ключевое изменение, которое устраняет дублирование и снижает риск ошибок при добавлении новых категорий. Всё остальное строится на этой основе.

**Independent Test**: Можно проверить независимо, добавив новую тестовую категорию через фабрику и убедившись, что она регистрируется корректно без написания повторяющегося кода.

**Acceptance Scenarios**:

1. **Given** есть новая категория навигации и список ключей, **When** разработчик вызывает `registerCategory(bot, category, keys, keyMapper)`, **Then** все кнопки этой категории регистрируются в боте с полной логикой (answerCbQuery, получение пользователя, pushToStack, track, выбор обработчика).
2. **Given** фабрика зарегистрировала обработчики, **When** пользователь нажимает кнопку, **Then** бот корректно отвечает, добавляет состояние в стек и фиксирует действие в истории.
3. **Given** разработчик добавляет новую категорию, **When** он использует фабрику вместо ручного дублирования, **Then** для добавления категории достаточно одного вызова функции.

---

### User Story 2 - Разделение ответственности по файлам (Priority: P2)

Как разработчик, я хочу найти нужный обработчик в предсказуемом месте, а не искать его в одном монолитном файле на 300+ строк.

**Why this priority**: Улучшает навигацию по коду и снижает когнитивную нагрузку при внесении изменений. Зависит от фабрики (P1).

**Independent Test**: Можно проверить, открыв директорию `handlers/` и убедившись, что каждый файл содержит только один класс обработчиков, и найдя любой обработчик менее чем за 30 секунд.

**Acceptance Scenarios**:

1. **Given** структура проекта обновлена, **When** разработчик хочет найти обработчик `/start`, **Then** он находится в выделенном файле стартового обработчика.
2. **Given** структура проекта обновлена, **When** разработчик хочет изменить логику кнопки "Назад", **Then** она находится в выделенном файле обработчика навигации назад.
3. **Given** директория `handlers/` создана, **When** разработчик открывает `handlers/index.ts`, **Then** видит единый список регистрации всех обработчиков.

---

### User Story 3 - Сохранение поведения бота (Priority: P3)

Как пользователь бота, я не должен замечать никаких изменений в поведении бота после рефакторинга.

**Why this priority**: Обязательное условие корректности рефакторинга. Проверяется независимо от структуры кода.

**Independent Test**: Можно проверить полностью независимо, запустив бота и протестировав все кнопки, команду `/start`, навигацию "Назад" и обработчики с изображениями.

**Acceptance Scenarios**:

1. **Given** бот запущен после рефакторинга, **When** пользователь отправляет `/start`, **Then** бот отвечает так же, как до рефакторинга.
2. **Given** пользователь находится на любом экране, **When** нажимает кнопку "Назад", **Then** бот возвращает на предыдущий экран через навигационный стек.
3. **Given** экран содержит изображение, **When** пользователь переходит на него, **Then** изображение отображается корректно.
4. **Given** пользователь отправляет неизвестное сообщение, **When** бот его получает, **Then** срабатывает fallback-обработчик, как раньше.

---

### Edge Cases

- Что происходит, если при рефакторинге какой-то импорт не обновлён? → TypeScript выдаёт ошибку компиляции, `npm run build` завершится с ошибкой.
- Что происходит, если фабрика вызвана с пустым списком ключей? → Обработчики не регистрируются, ошибок не возникает.
- Что происходит со старым файлом `handlers.ts`, если он ещё импортируется где-то? → Сборка выдаёт ошибку — файл должен быть полностью удалён и все импорты обновлены.
- Что происходит при ошибке внутри обработчика? → Ошибка логируется, выполнение не прерывает работу других обработчиков.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Система ДОЛЖНА предоставлять функцию-фабрику, которая инкапсулирует повторяющийся паттерн регистрации обработчиков кнопок.
- **SC-002**: Добавление новой категории обработчиков требует минимальных изменений:
  - новый файл модуля (при необходимости)
  - один вызов фабрики в оркестраторе
- **FR-003**: Внутри фабрики ДОЛЖНА быть сосредоточена вся повторяющаяся логика регистрации обработчиков, включая:
  - answerCbQuery
  - получение пользователя
  - pushToStack
  - track
  - выбор обработчика (с изображением / без)
  Фабрика НЕ ДОЛЖНА содержать бизнес-логику, специфичную для конкретной категории.
- **FR-004**: Обработчики ДОЛЖНЫ быть разбиты на отдельные файлы по ответственности: стартовый обработчик, обработчики групп, обработчик "Назад", обработчик админки, fallback-обработчики.
- **FR-005**: ДОЛЖЕН быть создан единый файл-оркестратор, который импортирует все обработчики и регистрирует их в одном месте.
- **FR-006**: Исходный монолитный файл обработчиков ДОЛЖЕН быть удалён, все его импорты — обновлены на новую структуру.
- **FR-007**: Поведение бота ДОЛЖНО остаться идентичным до и после рефакторинга: все кнопки, стек навигации, изображения, fallback.
- **FR-008**: Проект ДОЛЖЕН успешно собираться без ошибок после рефакторинга.
- **FR-009**: Обработчики НЕ ДОЛЖНЫ содержать бизнес-логику и ОБЯЗАНЫ только вызывать соответствующие функции или сервисы.

### Key Entities

- **Фабрика регистрации**: Функция, принимающая параметры категории и скрывающая всю повторяющуюся логику регистрации обработчиков кнопок.
- **Модули обработчиков**: Отдельные файлы, каждый из которых отвечает за одну категорию логики (старт, группы, назад, админка, fallback).
- **Оркестратор**: Единая точка входа, собирающая все обработчики и передающая их боту.
- **KeyMapper**: Функция-маппер, преобразующая ключ кнопки в параметры для отображения экрана.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Все кнопки Telegram-бота работают корректно после рефакторинга — 100% существующих сценариев проходят ручную проверку.
- **SC-002**: Добавление новой категории обработчиков требует изменений только в двух местах: новый файл модуля + одна строка в оркестраторе.
- **SC-003**: Объём дублирующегося кода регистрации обработчиков сокращён не менее чем на 60% по сравнению с исходным файлом.
- **SC-004**: Проект успешно компилируется без ошибок после рефакторинга.
- **SC-005**: Каждый файл в директории обработчиков содержит только одну категорию логики.

## Assumptions

- Рефакторинг затрагивает только внутреннюю структуру кода; пользовательский интерфейс бота не изменяется.
- Существующие типы для категорий навигации и маппинга ключей остаются без изменений и переиспользуются фабрикой.
- Ручная проверка поведения бота является достаточным способом подтверждения совместимости (автоматизированные тесты отсутствуют в проекте).
- Файлы со статическими данными навигации (`buttons.ts`, `buttonKeys.ts`) не изменяются.
- Модуль навигационного стека (выделен в рамках 001-navigation-stack-refactor) используется как есть.
