# Feature Specification: Типобезопасные ключи экранов и кнопок

**Feature Branch**: `004-typed-button-keys`
**Created**: 2026-03-30
**Status**: Draft
**Input**: User description: "Типобезопасные ключи экранов и кнопок — строгая типизация ScreenKey и ButtonKey, выведенная из buttonKeys"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Разработчик добавляет новый экран (Priority: P1)

Разработчик добавляет новую запись в `buttonKeys`. TypeScript автоматически подсвечивает все места в `buttons.ts` и `handlers.ts`, где нужно обработать новый ключ. После добавления обработчика проект компилируется без ошибок.

**Why this priority**: Это основной сценарий — добавление экранов происходит регулярно, и именно здесь типизация даёт наибольшую пользу.

**Independent Test**: Добавить тестовый ключ в `buttonKeys`, убедиться что компиляция падает с указанием мест, где нужно обновить код, затем добавить обработчик и убедиться что компиляция проходит.

**Acceptance Scenarios**:

1. **Given** новый ключ добавлен в `buttonKeys`, **When** запущена компиляция, **Then** TypeScript показывает ошибки во всех файлах, где ожидается обработка этого ключа
2. **Given** обработчик добавлен, **When** запущена компиляция, **Then** ошибок нет

---

### User Story 2 - Разработчик опечатывается в ключе (Priority: P2)

Разработчик случайно пишет несуществующий ключ экрана или кнопки в обработчике. TypeScript сразу подсвечивает ошибку — ключ не входит в допустимый набор.

**Why this priority**: Профилактика багов — основная мотивация для типизации.

**Independent Test**: Использовать несуществующий ключ (например, `screen_main` вместо `main`) в любом месте кода и убедиться, что компиляция падает.

**Acceptance Scenarios**:

1. **Given** в коде использован несуществующий ключ экрана, **When** запущена компиляция, **Then** TypeScript выдаёт ошибку с указанием недопустимого значения
2. **Given** в коде использован несуществующий ключ кнопки, **When** запущена компиляция, **Then** TypeScript выдаёт ошибку

---

### User Story 3 - Рассинхронизация buttonKeys и buttons (Priority: P3)

В `buttons.ts` определён экран, которого нет в `buttonKeys`. TypeScript ловит это на этапе компиляции.

**Why this priority**: Менее частый сценарий, но важный для целостности данных.

**Independent Test**: Добавить ключ в `buttons.ts`, отсутствующий в `buttonKeys`, и убедиться что компиляция падает.

**Acceptance Scenarios**:

1. **Given** в `buttons.ts` есть ключ, отсутствующий в `buttonKeys`, **When** запущена компиляция, **Then** TypeScript выдаёт ошибку о несоответствии типов

---

### Edge Cases

- Что происходит с динамическими ключами групп (регулярка `/^group_/`)? Они не типизируются статически и остаются `string`.
- Что происходит при пустом `buttonKeys`? Типы должны корректно выводиться как `never`.
- Ключи с одинаковыми значениями в разных массивах `buttonKeys` — `ButtonKey` не должен содержать дубликатов.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Объект `buttonKeys` ДОЛЖЕН быть помечен `as const` для вывода литеральных типов
- **FR-002**: Тип `ScreenKey` ДОЛЖЕН выводиться как `keyof typeof buttonKeys` — объединение всех ключей объекта `buttonKeys`
- **FR-003**: Тип `ButtonKey` ДОЛЖЕН выводиться как объединение всех значений массивов внутри `buttonKeys` (уникальных)
- **FR-004**: Объект `buttons` ДОЛЖЕН использовать тип `Record<ScreenKey | ButtonKey, ButtonRow[]>` вместо `Record<string, ButtonRow[]>`
- **FR-005**: Все места в `utils.ts` и `handlers.ts`, использующие `string` для ключей экранов или кнопок, ДОЛЖНЫ быть заменены на `ScreenKey` или `ButtonKey`
- **FR-006**: Компиляция ДОЛЖНА завершаться с ошибкой при использовании несуществующего ключа экрана или кнопки
- **FR-007**: Компиляция ДОЛЖНА завершаться с ошибкой при наличии ключа в `buttons.ts`, отсутствующего в `buttonKeys`
- **FR-008**: Динамические ключи групп (шаблон `group_*`) ДОЛЖНЫ остаться строковыми — статическая типизация к ним не применяется

### Key Entities

- **ScreenKey**: Литеральный тип — объединение всех ключей объекта `buttonKeys`. Источник истины для идентификаторов экранов.
- **ButtonKey**: Литеральный тип — объединение всех значений массивов `buttonKeys`. Идентификаторы кнопок навигации.
- **buttonKeys**: Константный объект (`as const`), определяющий структуру навигации: каждому экрану соответствует массив кнопок-потомков.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Проект компилируется без ошибок после типизации — существующая функциональность не нарушена
- **SC-002**: Опечатка в ключе экрана или кнопки обнаруживается на этапе компиляции, а не в рантайме
- **SC-003**: Все `string`-типы для навигационных ключей заменены на `ScreenKey` / `ButtonKey` (кроме динамических групп)
- **SC-004**: При добавлении нового экрана в `buttonKeys` разработчик получает автоподсказки IDE по всем местам, где нужно обновить код

## Assumptions

- Типы определяются единственным источником истины — файлом `buttonKeys.ts`
- Ручные union-типы не используются — только вывод из `buttonKeys` через indexed access types
- Динамические группы (ключи вида `group_*`) не типизируются — они создаются через админ-панель и обрабатываются regex-обработчиком
- Файл `buttonKeys.ts` содержит корректные данные, соответствующие текущей навигации бота
- Обратная совместимость: runtime-поведение бота не меняется, изменения только в типах
