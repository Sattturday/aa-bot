# Исследование: Типизация buttonKeys

## Решение 1: Где определить типы ScreenKey и ButtonKey

**Решение**: Типы определяются в `src/data/buttonKeys.ts` рядом с объектом-источником.

**Обоснование**: `buttonKeys` — единственный источник истины. Типы должны выводиться из него и экспортироваться рядом. Импорт из другого файла создал бы ненужную зависимость.

**Альтернативы**:
- Отдельный файл `src/types/navigation.ts` — отвергнуто: лишняя косвенность, нарушение принципа «один источник истины»
- `src/types/index.ts` — отвергнуто по той же причине

## Решение 2: Как вывести ButtonKey (union значений массивов)

**Решение**: Использовать `typeof buttonKeys[keyof typeof buttonKeys][number]`.

```ts
type ButtonKey = typeof buttonKeys[keyof typeof buttonKeys][number];
```

**Обоснование**: Стандартный паттерн TypeScript для union значений из `as const` объекта с массивами. Работает без утилит-типов.

**Альтернативы**:
- Ручной union тип — отвергнуто: нужно поддерживать вручную при каждом изменении
- `ValuesOf<typeof buttonKeys>` — кастомный утилит-тип, избыточен для одной строки

## Решение 3: Несовпадение ключей между buttonKeys и buttons

**Решение**: Некоторые экраны определены в `buttons.ts`, но не в `buttonKeys` (например, `literature`, `ask_question`). Также `buttonKeys` содержит `group_schedule` с группами, которых нет в `buttons.ts` (генерируются динамически).

Стратегия:
- `buttons.ts` типизировать как `Record<ScreenKey | ButtonKey | 'back', ButtonRow[]>` — ключ `back` используется повсеместно, но не входит ни в один массив
- `group_schedule` — это `ScreenKey`, но кнопки генерируются динамически. В `buttons.ts` его нет, но это допустимо (runtime-проверка в `utils.ts` уже обрабатывает это)
- Ключи вроде `literature`, `ask_question` — экраны без записи в `buttonKeys`. Нужно либо добавить их в `buttonKeys`, либо расширить тип `buttons`

**Действие**: Добавить недостающие ключи экранов в `buttonKeys` с пустыми массивами кнопок-потомков, чтобы типы были полными.

**Обоснование**: Конституция V — минимальная сложность. Проще дополнить `buttonKeys` как источник истины, чем вводить отдельные union-типы.

## Решение 4: navigationStack — string или ScreenKey?

**Решение**: Оставить `state: string` в `pushToStack` / `popFromStack`.

**Обоснование**: Навигационный стек хранит произвольные ключи экранов, включая динамические группы (`group_aa_moscow`). Типизация стека как `ScreenKey` исключила бы динамические ключи и потребовала бы union с `string`. Это не даёт пользы — стек не валидирует ключи.

## Решение 5: Ключ 'back' в кнопках

**Решение**: `'back'` не входит ни в один массив `buttonKeys`, но используется в большинстве экранов `buttons.ts`. Добавлять в типы не нужно — достаточно расширить тип `buttons`:

```ts
Record<ScreenKey | ButtonKey, ButtonRow[]>
```

Кнопка `back` — это callback-данные, а не ключ экрана. Обработчик `back` регистрируется отдельно в `handlers/back.ts` и использует `popFromStack`.

**Альтернатива**: Добавить `'back'` в union — избыточно, это не самостоятельный экран.

## Решение 6: factory.ts — типы ключей

**Решение**: Параметр `category` в `RegisterCategoryOptions` сделать `ScreenKey`, `keys` — `readonly ButtonKey[]`, `keyMapper` — `(key: ButtonKey) => KeyMapResult`.

**Обоснование**: Это максимально строгая типизация, совместимая с текущим использованием. Каждый handler-файл передаёт конкретный `category` (литерал) и `buttonKeys.xxx` (массив литералов).
