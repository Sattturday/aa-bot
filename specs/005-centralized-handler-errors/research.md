# Исследование: Централизованная обработка ошибок в Telegram-хендлерах

## Решение 1: Где размещать единый error-wrapper

**Решение**: Разместить обёртку в отдельном модуле `src/utils/handlers/withErrorHandler.ts`.

**Обоснование**: Ошибка обрабатывается одинаково минимум в `factory.ts`, `start.ts`, `groups.ts`, `back.ts`, `admin.ts` и `fallback.ts`. Это удовлетворяет принципу повторного использования и не смешивает инфраструктурную логику с доменной.

**Альтернативы**:
- Оставить `try/catch` в каждом хендлере и лишь унифицировать текст сообщения. Отвергнуто: дублирование сохранится.
- Перенести логику в `src/handlers/factory.ts`. Отвергнуто: wrapper нужен не только фабричным хендлерам.

## Решение 2: Что именно должна делать обёртка

**Решение**: Wrapper принимает label и async-хендлер, выполняет исходный код внутри общего `try/catch`, логирует ошибку через `console.error`, затем один раз пытается отправить пользователю fallback-сообщение.

**Обоснование**: Это покрывает пользовательское требование единого UX и техническое требование единого журнала ошибок, не меняя основной поток успешного сценария.

**Альтернативы**:
- Логировать без ответа пользователю. Отвергнуто: не выполняет P1-ценность из спецификации.
- Отправлять разные fallback-сообщения по типам ошибок. Отвергнуто: выходит за scope текущей фичи.

## Решение 3: Как обрабатывать вторичную ошибку при fallback-ответе

**Решение**: Ошибку при `ctx.reply`, `ctx.replyWithPhoto`, `ctx.answerCbQuery` или другой операции, используемой для fallback, логировать отдельно и проглатывать без повторного выброса.

**Обоснование**: Спецификация требует, чтобы сбой fallback-пути не ломал дальнейшую работу бота. Повторный throw привёл бы к рекурсивному или каскадному падению.

**Альтернативы**:
- Повторно выбрасывать fallback-ошибку выше. Отвергнуто: нарушает FR-005.
- Молчаливо игнорировать без логирования. Отвергнуто: осложняет диагностику инцидентов.

## Решение 4: На каком уровне подключать wrapper

**Решение**: Оборачивать функции прямо в местах регистрации Telegram-хендлеров: `bot.action`, `bot.start`, `bot.on`, regex-action и внутри фабрики `registerCategory`.

**Обоснование**: Уровень регистрации является единой точкой входа для Telegraf. Так можно убрать локальные `try/catch` из хендлеров и не менять сигнатуры нижележащих утилит без необходимости.

**Альтернативы**:
- Оборачивать только вспомогательные `handleButtonAction*` функции. Отвергнуто: не покрывает `admin`, `fallback`, `back` и другие прямые хендлеры.
- Использовать Telegraf middleware глобально. Отвергнуто: более широкое изменение, чем просит задача, и сложнее привязать label к конкретному сценарию.

## Решение 5: Какие данные включать в лог ошибки

**Решение**: Логировать label, исходный объект ошибки и `userId`, если он доступен из `ctx.from`.

**Обоснование**: Этого достаточно, чтобы понимать точку сбоя и затронутого пользователя, без избыточного логирования полного контекста Telegram update.

**Альтернативы**:
- Логировать весь `ctx.update`. Отвергнуто: шумно и потенциально избыточно.
- Логировать только message string. Отвергнуто: теряется stack и контекст.

## Решение 6: Как закрыть требование конституции по тестам

**Решение**: В рамках реализации добавить минимальный root `npm test` и unit-тесты для wrapper-логики, включая основной error-path и вторичный fallback-сбой.

**Обоснование**: Конституция IX требует запуск `npm test` после каждого изменения и наличие тестов для бизнес-критичных сценариев. Хотя проект пока без тестов, именно эта фича напрямую меняет поведение при ошибках и требует проверяемого safety net.

**Альтернативы**:
- Ограничиться ручной проверкой. Отвергнуто: противоречит конституции.
- Перенести задачу тестов в отдельную фичу. Отвергнуто: сделало бы текущий rollout не соответствующим проектным правилам.
