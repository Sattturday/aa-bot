# Feature Specification: Централизованная обработка ошибок в Telegram-хендлерах

**Feature Branch**: `005-centralized-handler-errors`
**Created**: 2026-03-31
**Status**: Draft
**Input**: User description: "@.specify/prompts/005-specify_prompt.md"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Понятное сообщение при сбое (Priority: P1)

Как пользователь Telegram-бота, я хочу получать понятное сообщение, если действие не удалось, чтобы понимать, что произошёл сбой и как продолжить работу.

**Why this priority**: Это основной пользовательский эффект. Без него изменение не даёт ценности.

**Independent Test**: Искусственно вызвать сбой в любом обработчике и проверить, что пользователь получает единое сообщение.

**Acceptance Scenarios**:

1. **Given** пользователь инициирует действие, **When** в обработчике происходит ошибка, **Then** бот отправляет единое сообщение об ошибке.
2. **Given** произошёл сбой, **When** пользователь получает сообщение, **Then** оно содержит рекомендацию повторить действие или использовать `/start`.
3. **Given** произошёл сбой, **When** обработка завершена, **Then** бот остаётся работоспособным.

---

### User Story 2 - Единое поведение ошибок во всех обработчиках (Priority: P2)

Как разработчик, я хочу единый механизм обработки ошибок, чтобы не дублировать `try-catch` и не получать разное поведение при сбоях.

**Why this priority**: Уменьшает дублирование и снижает риск пропуска обработки ошибок.

**Independent Test**: Вызвать ошибки в разных обработчиках и убедиться, что:

* логирование одинаковое
* сообщение пользователю одинаковое

**Acceptance Scenarios**:

1. **Given** разные обработчики, **When** в них происходит ошибка, **Then** поведение идентично.
2. **Given** добавлен новый обработчик, **When** он зарегистрирован, **Then** дополнительная логика обработки ошибок не требуется.

---

### User Story 3 - Отсутствие регрессий (Priority: P3)

Как пользователь, я хочу, чтобы при отсутствии ошибок всё работало как раньше.

**Why this priority**: Это рефакторинг, он не должен ломать текущий UX.

**Independent Test**: Пройти основные сценарии (`/start`, кнопки, навигация) без ошибок.

**Acceptance Scenarios**:

1. **Given** сценарий без ошибок, **When** пользователь взаимодействует с ботом, **Then** поведение не меняется.
2. **Given** навигация назад, **When** ошибок нет, **Then** стек работает как раньше.

---

### Edge Cases

* Ошибка при отправке fallback-сообщения → игнорируется, бот не падает
* Ошибка до формирования ответа → пользователь всё равно получает fallback (если возможно)
* Ошибка в любом типе обработчика (action, command, text) → обрабатывается единым механизмом
* Контекст (`ctx`) недоступен или ответ невозможен → ошибка логируется, повторных попыток нет
* Асинхронная операция без `await` → не гарантируется перехват (ограничение системы)

## Requirements *(mandatory)*

### Functional Requirements

* **FR-001**: Система ДОЛЖНА обрабатывать ошибки одинаково во всех точках входа Telegram-бота.
* **FR-002**: При любом необработанном сбое система ДОЛЖНА отправлять сообщение:
  `Произошла ошибка. Попробуйте ещё раз или нажмите /start`
* **FR-003**: Система ДОЛЖНА логировать каждую ошибку через единый механизм.
* **FR-004**: Хендлеры НЕ ДОЛЖНЫ содержать локальную стандартную обработку ошибок (`try-catch` с fallback).
* **FR-005**: Ошибка при отправке fallback-сообщения НЕ ДОЛЖНА прерывать выполнение.
* **FR-006**: Механизм ДОЛЖЕН применяться ко всем обработчикам независимо от способа регистрации.
* **FR-007**: При отсутствии ошибок поведение системы НЕ ДОЛЖНО изменяться.
* **FR-008**: Изменение ДОЛЖНО покрывать все существующие пользовательские сценарии.

---

### Архитектурные требования

* **FR-009**: Обработка ошибок ДОЛЖНА применяться на уровне регистрации хендлеров (через фабрику и централизованные точки подключения).
* **FR-010**: Все асинхронные операции внутри хендлеров ДОЛЖНЫ быть awaited или явно обработаны.
* **FR-011**: Лог ошибки ДОЛЖЕН содержать:

  * label (контекст)
  * ошибку (message/stack)
  * userId (если доступен)
* **FR-012**: Подключение обработки ошибок ДОЛЖНО выполняться одной обёрткой без изменения бизнес-логики.

---

### Точки входа (область покрытия)

Механизм ДОЛЖЕН применяться к:

* `bot.action(...)`
* `bot.command(...)`
* `bot.on(...)` (text, message и др.)
* fallback-хендлерам

---

## Key Entities

* **Пользовательский сценарий**: любое действие пользователя, вызывающее обработчик
* **Контекст ошибки (label)**: строка, описывающая место возникновения ошибки
* **Fallback-сообщение**: стандартный ответ пользователю при сбое
* **Точка входа**: место регистрации обработчика в Telegram-боте

---

## Success Criteria *(mandatory)*

### Measurable Outcomes

* **SC-001**: В 100% хендлеров при сбое пользователь получает одинаковое сообщение
* **SC-002**: В 100% случаев ошибка логируется с контекстом
* **SC-003**: В коде отсутствуют локальные `try-catch`, реализующие стандартное fallback-поведение
* **SC-004**: Все текущие сценарии без ошибок работают без изменений
* **SC-005**: Ошибка при отправке fallback-сообщения не вызывает повторного падения

---

## Assumptions

* Все ошибки в рамках фичи считаются техническими и обрабатываются единым сообщением
* Разделение типов ошибок (validation, бизнес-ошибки) не входит в текущий scope
* Используется текущий механизм логирования (`console.error`), но в централизованной форме
* Проверка корректности выполняется вручную (автотесты отсутствуют)
